# 后端子代理分工计划

## 1. 目的

后端开发阶段固定保留三个专项子代理，用于持续做并行检查，减少“功能做完后才发现安全/质量/测试缺口”的返工。

## 2. 子代理划分

### 安全检验控制代理

职责：

- 审查认证、RBAC、企业数据隔离、文件安全、导出安全
- 审查审计日志与状态机约束是否完整
- 审查配置与密钥管理、环境差异和限流策略

输出：

- 安全控制清单
- 高风险越权点
- 必补的审计点
- 安全回归检查项

### 代码质量审查代理

职责：

- 审查包结构、分层边界、命名规范
- 审查 DTO / VO / Entity 边界
- 审查异常处理、日志、事务边界、枚举设计
- 审查静态检查、覆盖率和 OpenAPI 维护方式

输出：

- 代码质量规则
- PR 合并门槛
- 工具链建议
- 分层违规提醒

### 功能测试代理

职责：

- 设计关键业务闭环测试
- 审查状态流转、权限隔离、导入、文件、消息测试覆盖
- 给出 P0/P1/P2 回归优先级

输出：

- 最小可执行测试集
- 集成测试清单
- 回归测试矩阵

## 3. 主代理与子代理协作方式

主代理负责：

- 定路线
- 改代码
- 做集成
- 跑最终验证

子代理负责：

- 专项并行审查
- 输出高风险提醒
- 不直接修改主干方案的边界定义

## 4. 触发规则

以下情况默认触发安全代理：

- 登录、注册、验证码、权限、导出、上传、审核流

以下情况默认触发质量代理：

- 新增模块骨架
- 新增统一基础设施
- 批量提交较多 Java 文件

以下情况默认触发测试代理：

- 新增完整业务链路
- 状态机修改
- 导入导出与文件能力新增

## 5. 输出落地位置

- 质量建议：沉淀到 `pom.xml`、编码规范、PR 门槛
- 测试建议：沉淀到 `docs/kickoff/05-backend-verification-checklist.md` 与 `backend/src/test`
