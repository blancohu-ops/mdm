# 权限体系专项实施计划

## 1. 目标

本计划用于指导“工业企业出海主数据平台”一期权限体系的设计落地与开发实施。

一期目标不是一次性做成全量策略引擎，而是在现有 `enterprise_owner / reviewer / operations_admin`
基础上，升级为一套可落地、可审计、可扩展的权限体系，覆盖以下核心问题：

- 平台侧与企业侧租户隔离
- 角色权限与数据范围解耦
- 高风险动作单独受控
- AI 工具权限单独建模
- 权限变更、导出、审批、AI 调用可审计

## 2. 设计原则

- 租户隔离优先，所有企业核心数据默认按 `tenant` 强隔离
- 一期采用“`RBAC` 为主、数据范围规则为辅、高风险能力单列”的渐进式方案
- 菜单和页面只负责展示，最终准入以服务端接口和数据层校验为准
- 角色只承载稳定职责，不承载临时范围和高风险能力
- 数据范围统一收敛，不在一期引入大面积字段级权限
- AI 权限单独建模，不继承“能看就能问、能问就能写”
- 所有允许和拒绝都应能解释生效依据

## 3. 一期范围

### 3.1 纳入一期

- 权限对象模型：用户、租户、组织、角色模板、能力包、资源、动作、数据范围
- 后端统一授权判定入口
- 企业侧与平台侧角色模板
- 数据范围规则：`SELF / ORG / TENANT / ASSIGNED_DOMAIN / TEMP_GRANTED`
- 高风险能力包：敏感导出、跨租户支援、发布、角色授权、AI 高级能力、应急访问
- 临时授权与自动回收
- 高风险动作审计
- AI 工具分级授权

### 3.2 暂不纳入一期

- 全量字段级权限
- 通用策略 DSL / 可视化策略引擎
- 外部合作伙伴复杂共享模型
- 对外开放 API 授权体系
- 完整 ReBAC 平台化落地

## 4. 权限模型基线

### 4.1 权限判定公式

`允许 = 租户边界通过 × 角色功能通过 × 数据范围通过 × 高风险控制通过`

### 4.2 建议权限点命名

- 统一使用 `resource:action`
- 例如：
  - `enterprise_profile:read`
  - `enterprise_application:approve`
  - `product:submit`
  - `file_asset:download`
  - `ai_tool:ask_ai`
  - `role_template:grant`

### 4.3 数据范围

- `SELF`
- `ORG`
- `TENANT`
- `ASSIGNED_DOMAIN`
- `TEMP_GRANTED`

### 4.4 附加能力包

- `sensitive_export`
- `cross_tenant_support`
- `content_publish`
- `role_grant`
- `ai_advanced`
- `break_glass`

## 5. 当前实现中的优先风险

以下风险建议在权限体系开发前半段优先处理：

1. 文件访问控制不够严格，存在对象级越权风险
2. 管理端当前以粗粒度角色放行，缺少分配范围、区域、行业等数据范围控制
3. 审计闭环不足，缺少权限变更、高风险导出、AI 调用的独立审计
4. 开发态认证配置仍较多，需避免默认值和 Mock 逻辑进入非开发环境
5. 前端 `localStorage` 持有令牌，后续要评估更稳妥的会话方案

## 6. 分阶段开发计划

### 阶段 0：冻结规则与补地基

预计：2-3 个工作日

目标：

- 固化权限对象、权限点命名、数据范围、能力包清单
- 识别并补齐当前最危险的越权点
- 搭建权限专项测试骨架

主要任务：

- 输出平台侧/企业侧角色模板草案
- 输出权限点清单与数据范围规则表
- 修补文件访问控制的对象级授权缺口
- 梳理开发/测试/生产配置边界，避免 Mock 与默认安全配置误入非开发环境
- 建立后端权限测试样例数据与基础测试类

交付物：

- 权限对象清单
- 角色模板草案
- 权限点字典
- 数据范围规则表
- 高风险风险点修复清单
- 测试骨架与样例数据方案

退出条件：

- 权限口径完成一次评审
- 文件访问高危越权点收口
- 后端权限测试可开始编写

### 阶段 1：权限内核与数据模型

预计：3-4 个工作日

目标：

- 建立统一授权内核
- 完成角色模板、权限点、能力包、范围规则的数据模型

主要任务：

- 新增权限核心模块，建议独立为 `iam` 或同等级公共安全模块
- 建立以下核心对象：
  - 角色模板
  - 权限点
  - 角色-权限映射
  - 用户-角色绑定
  - 用户-能力包绑定
  - 数据范围规则
  - 授权判定结果与拒绝原因模型
- 在服务端形成统一授权入口，避免各业务模块散落写死
- 保留现有 3 个基础角色，并映射到新模板体系

交付物：

- 权限核心表结构 / 实体 / 迁移脚本
- 授权服务接口
- 角色与能力包装配逻辑
- 拒绝原因返回模型

退出条件：

- 至少支持 `resource:action + scope` 的统一判定
- 现有企业侧/平台侧角色可映射到新体系

### 阶段 2：业务模块接入

预计：4-6 个工作日

目标：

- 将权限内核接入现有关键业务模块
- 把“角色放行”升级为“角色 + 范围 + 对象归属”判定

优先接入顺序：

1. `file`
2. `enterprise`
3. `enterpriseReview`
4. `product`
5. `productReview`
6. `importtask`
7. `category`
8. `message`

主要任务：

- 企业侧接口全部补齐 `tenant` 与对象归属校验
- 平台侧审核接口接入 `ASSIGNED_DOMAIN` 范围控制
- 文件、下载、导入、导出按对象归属和能力包做强校验
- 前端接入页面显隐、菜单显隐、按钮显隐，但不将前端作为安全边界

交付物：

- 模块级权限接入清单
- 平台侧/企业侧角色矩阵
- 前后端权限映射表

退出条件：

- 关键模块不再只依赖粗粒度角色判断
- 跨租户访问与对象越权访问有明确拒绝逻辑

### 阶段 3：高风险动作、审批与 AI 权限

预计：3-4 个工作日

目标：

- 将高风险动作从普通查看/编辑权限中独立出来
- 为 AI 工具建立单独权限层

主要任务：

- 落地临时授权与自动过期
- 落地敏感导出审批与审计
- 落地跨租户支援审批
- 落地内容发布与角色授权的高风险控制
- AI 权限拆分为：
  - `ask_ai`
  - `generate_ai`
  - `export`
  - `writeback_ai`
- AI 检索范围走服务端授权过滤
- 记录 AI 请求、检索命中、工具调用、结果摘要审计

交付物：

- 临时授权机制
- 高风险审批流骨架
- AI 分级权限与审计逻辑

退出条件：

- 高风险动作可单独控制与追踪
- AI 权限不再复用普通查看权限

### 阶段 4：测试、回归与发布准备

预计：3-4 个工作日

目标：

- 为权限体系建立可持续的回归验证能力
- 将权限变更纳入 CI 基线

主要任务：

- 建立权限内核单元测试
- 建立后端接口集成测试：
  - `401 / 403`
  - 跨租户拒绝
  - 范围差异返回不同数据
  - query/path/body 篡改无法越权
- 建立高风险动作与审计测试
- 建立 AI 权限专项测试
- 扩充前端权限体验测试与关键 E2E 流程
- 将专项用例纳入 CI 必跑集

交付物：

- 权限矩阵测试集
- 高风险动作回归集
- AI 权限测试集
- 更新后的验证清单

退出条件：

- 关键权限路径具备自动化回归能力
- 开发、测试、评审均有统一准入标准

## 7. 子代理协作安排

### 安全线

重点负责：

- 租户隔离
- 对象级授权
- 文件安全
- 导出与临时授权安全
- AI 越权与审计边界

默认介入场景：

- 认证、授权、导出、上传、审批流、AI 工具

### 代码质量线

重点负责：

- 权限内核模块分层
- 命名与权限点规范
- 角色、范围、能力包职责边界
- 避免权限判断散落在 Controller 和前端

默认介入场景：

- 新增权限模块
- 批量修改业务模块鉴权
- 公共基础设施抽象

### 测试线

重点负责：

- 允许/拒绝矩阵
- 角色与范围测试
- 跨租户与越权回归
- 高风险动作审计验证
- AI 权限专项测试

默认介入场景：

- 状态流变更
- 审批流变更
- 导入导出能力新增
- AI 权限能力新增

## 8. 开发顺序建议

建议按以下节奏推进：

1. 先补高危安全缺口与冻结规则
2. 再做权限内核与数据模型
3. 再接关键业务模块
4. 再接审批、审计与 AI
5. 最后扩大测试面并收口

不建议的顺序：

- 一开始就做前端菜单权限
- 一开始就上完整策略引擎
- 一开始就做全量字段级权限
- 在缺少测试骨架时直接大面积改业务模块

## 9. 完成定义

权限体系一期完成，至少应满足以下标准：

- 平台侧与企业侧角色边界清晰
- 企业核心数据、产品、文件、消息具备后端强隔离
- 审核与管理动作具备范围控制
- 导出、跨租户、授权、AI 写回属于高风险动作并有独立控制
- 审计能回答“谁在什么时候因为什么规则访问了什么资源”
- 关键允许与拒绝场景已纳入自动化验证

## 10. 当前仍缺的输入

以下信息建议在阶段 0 评审中明确：

- 企业内部组织层级是否一期就要落地
- 平台审核员的分配维度是区域、行业、类目还是企业池
- 哪些字段必须视为敏感字段
- 哪些导出动作属于默认禁止
- AI 工具一期具体开放哪些能力
- 应急访问是否需要双人审批

## 11. 执行入口

建议从“阶段 0：冻结规则与补地基”开始实施，并将以下命令作为基本验证入口：

- 前端构建：`npm run build`
- 前端 E2E：`npm run test:e2e`
- 后端测试：`mvn -f backend/pom.xml test`

阶段 0 完成后，再正式进入权限内核开发。
